MTS Finance Dashboard - Current State Design & Technical Specification

Version 2.1 | Date: 2026-03-09 | Audience: Business Owner, UI Engineers, Backend Engineers, AI Coding Agents

Status: Active implementation UI Stack: HTML/CSS/Vanilla JavaScript Backend: Spring Boot 4 + Java 21 + Oracle Target user: non-technical business operator

1. Product Objective

Provide a simple daily-use dashboard for purchase management in a small business setup, minimizing manual dependency and reducing data entry mistakes.

  • Maintain master data (sellers, units, product types, products, variants).
  • Create and manage purchase orders using draft-to-confirm flow.
  • Enable one-click finalization for draft purchase orders.
  • Provide reporting views for monthly and period-based analysis.

2. Current Functional Scope (Implemented)

2.1 Authentication and Session

  • Login page with username/password authentication.
  • Session token-based flow with backend validation.
  • Configured 15-minute session expiry design.
  • Logout action available in top navigation as icon button with tooltip.

2.2 Landing Page

  • Create Purchase Order form with header fields and dynamic line items.
  • Dependent flow in item row: Product -> Variant -> Unit (auto).
  • Line total auto-calculation based on quantity and rate.
  • Draft creation and optional immediate confirm flow.
  • Current Month Purchase History table with status badges:
    • CONFIRMED green
    • DRAFT yellow
  • History actions on draft rows: Edit, Confirm, Delete.
  • Draft edit popup supports both Save Draft and Save & Confirm.
  • Invoice column removed from history table (latest UX update).
  • Product-type-wise current month pie chart shown on landing.

2.3 Maintenance Page

  • Master management sections for Sellers, Units, Product Types, Products, and Variants.
  • Variant creation under selected product.
  • Unit input for variant changed from manual ID to dropdown selection.
  • Product list includes product-wise variant count.

2.4 Reporting Page

  • Graph 1: Daily purchase trend (date range with backend validation).
  • Graph 2: Top products by units + variant drill-down on click.
  • Graph 3: Top sellers by purchase value.
  • Graph 2 and Graph 3 support top-5 visible with scroll behavior.
  • Reporting data includes CONFIRMED orders only.

3. UX/UI Design Decisions (Current)

  1. UI language is business-friendly and action-focused (non-technical audience).
  2. Navigation includes pages: Landing, Maintenance, Reporting, plus user chip and logout icon.
  3. Draft-first PO flow retained to support corrections before finalization.
  4. One-click confirm is available after initial insert and also from draft history/edit popup.
  5. Mobile responsiveness is actively optimized (form stacking, chart containment, button wrapping).

4. Backend API Contract Snapshot

4.1 Auth APIs

  • POST /api/auth/setup
  • POST /api/auth/login
  • GET /api/auth/me
  • POST /api/auth/logout

4.2 Master Data APIs

  • /api/sellers, /api/sellers/all
  • /api/units
  • /api/product-types
  • /api/products (supports includeVariants=true for one-call maintenance view)
  • GET /api/products/{id}/variants
  • POST /api/products/{id}/variants
  • GET /api/variants/{variantId} (used by draft edit flow)
  • PUT/DELETE /api/variants/{variantId}

4.3 Purchase Order APIs

  • GET /api/purchase-orders (date/seller filters)
  • GET /api/purchase-orders/{id}
  • POST /api/purchase-orders (create draft)
  • PUT /api/purchase-orders/{id} (update draft)
  • PATCH /api/purchase-orders/{id}/confirm
  • DELETE /api/purchase-orders/{id} (draft only)

4.4 Report APIs

  • GET /api/reports/trends/daily-purchase
  • GET /api/reports/trends/top-products
  • GET /api/reports/trends/top-sellers
  • GET /api/reports/daily
  • GET /api/reports/seller/{id}

5. Data & Business Rules

  • Purchase orders begin as DRAFT, then move to CONFIRMED.
  • Only confirmed orders are used in reporting aggregates.
  • Variant is attached to product; PO line persists variantId + quantity/rate.
  • Variant dropdown is product-filtered; unit is derived automatically.
  • Seller and product masters support active/inactive lifecycle.

6. Mobile & Responsiveness (Current Baseline)

  • Single-column layout for forms/charts on smaller screens.
  • Navigation links wrap and remain tappable.
  • Pie chart and total card constrained to avoid horizontal overflow.
  • History table remains scrollable horizontally when required.
Safari mobile cache can keep stale CSS/JS. Use hard refresh or clear website data after UI updates.

7. Environment & Deployment Configuration

7.1 Required Environment Variables

  • PROJ_ENVIRONMENT (current profile variable used by application.properties)
  • DB_URL
  • DB_USERNAME
  • DB_PASSWORD
  • AUTH_REG_OTP_FROM_EMAIL and AUTH_REG_OTP_OWNER_EMAIL
  • AUTH_TRANSPORT_REQUIRE_HTTPS=true for UAT/PROD
  • APP_CORS_ALLOWED_ORIGINS with trusted HTTPS UI origins only

7.2 VS Code Local Run

  • .env file is used through .vscode/launch.json.
  • .env.example is committed as a template.

7.3 Security Notes

  • Do not commit OCI wallet files to GitHub.
  • Do not commit real credentials in .env or property files.
  • If credentials were exposed, rotate DB password immediately.

8. Security Features and Required Controls

8.1 Implemented Security Controls

  • Auth endpoints enforce HTTPS transport in UAT/PROD using proxy-aware checks (X-Forwarded-Proto).
  • Frontend blocks insecure auth and authenticated API calls when page/API transport is not HTTPS.
  • Frontend API base override attack surface reduced by disabling URL/localStorage API base overrides.
  • CORS moved to allowlist model via app.cors.allowed-origins (wildcard origins removed).
  • Passwords are stored as BCrypt hashes; session tokens and OTPs are stored as hashes.
  • OTP registration flow has TTL and max-attempt limits; owner email is externalized to environment config.

8.2 Required Deployment Settings (for current architecture)

  • Nginx must terminate TLS on port 443 and redirect port 80 traffic to HTTPS.
  • Nginx must send Host, X-Forwarded-For, and X-Forwarded-Proto headers to backend.
  • Backend runtime env must set AUTH_TRANSPORT_REQUIRE_HTTPS=true and correct APP_CORS_ALLOWED_ORIGINS.
  • Self-signed certificates are acceptable only as temporary measure; browsers will show trust warning until CA-signed cert is used.

9. Observability and Error Handling

  • Global exception handling returns consistent API error envelopes.
  • Request logging filter logs entry and exit for all /api/** endpoints at INFO level.
  • Logs include method, path, status, duration, and user (when authenticated).

10. Known Gaps / Next Recommended Enhancements

  1. Replace temporary self-signed cert with CA-signed trusted cert (Let’s Encrypt/ACM/other).
  2. Add login and OTP rate limiting (Nginx and/or backend) to reduce brute-force risk.
  3. Protect or disable Swagger/OpenAPI endpoints in production environments.
  4. Add security response headers at nginx layer (HSTS, CSP, X-Content-Type-Options, Referrer-Policy).
  5. If dashboard remains in a separate repository/folder, maintain a dedicated UI release document and versioning.
